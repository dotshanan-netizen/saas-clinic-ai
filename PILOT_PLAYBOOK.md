# Clinova AI: Internal Pilot Playbook

هذه الوثيقة هي الدليل التشغيلي الرسمي لإجراء **الـ Pilot الداخلي (Internal Pilot Simulation)** قبل فتح النظام للعملاء الحقيقيين.

الهدف: التحقق من استقرار ودورة حياة النظام بالكامل E2E في بيئة محاكاة واقعية.

---

## 📋 خطوات سيناريو المحاكاة (Step-by-Step Scenario)

### الخطوة 1: تهيئة عيادة تجريبية جديدة (Onboarding)
قم بتشغيل معالج إعداد العيادات التفاعلي لإنشاء عيادة محاكاة:
```bash
npm run onboard-tenant
```
**البيانات المقترحة للإدخال:**
- **الاسم:** عيادة الابتسامة التجريبية
- **الرابط (Slug):** `smile-pilot`
- **معلومات الواتساب:** ضع قيم اختبارية (مثلاً `12345`)

*التحقق:* تأكد في قاعدة البيانات (أو عبر الكونسول) من إنشاء العيادة، الفرع الرئيسي، و12 ساعة عمل للملف الشخصي للعيادة بنجاح.

---

### الخطوة 2: تشغيل البنية التحتية بالكامل
تأكد من تشغيل كافة الخوادم في محيط التطوير أو الإنتاج:
1. **قاعدة البيانات والـ Redis:** يجب أن تكون متصلة.
2. **سيرفر الويب (Next.js):**
   ```bash
   npm run dev
   ```
3. **الـ Queue Worker (الخلفية):**
   ```bash
   npm run worker
   ```

---

### الخطوة 3: تسجيل الدخول الآمن للوحة التحكم
1. افتح المتصفح على الرابط: `http://localhost:3000/login`
2. أدخل البريد الإلكتروني وكلمة المرور لمدير العيادة الذي قمت بإنشائه في الخطوة 1.
3. اضغط على زر "دخول اللوحة 🚀".
4. *التحقق:* يجب أن تنجح عملية الدخول وتوجيهك لصفحة الـ Dashboard `/dashboard` الخاصة بالعيادة.

---

### الخطوة 4: محاكاة محادثة مريض (WhatsApp Message Simulation)
لتجنب استهلاك رصيد WhatsApp الفعلي، سنقوم بمحاكاة طلب POST إلى الـ Webhook مباشرة باستخدام `curl` أو سكريبت اختبار:

```bash
curl -X POST http://localhost:3000/api/webhook/whatsapp \
  -H "Content-Type: application/json" \
  -d '{
    "entry": [{
      "changes": [{
        "value": {
          "messages": [{
            "id": "wamid.test_message_pilot_1",
            "from": "966500000001",
            "text": { "body": "أهلاً، أبي أحجز موعد لتنظيف البشرة عند الأخصائية نورة فرع الصحافة بكرا العصر" },
            "type": "text"
          }],
          "metadata": {
            "phone_number_id": "12345"
          }
        },
        "field": "messages"
      }]
    }]
  }'
```

---

### الخطوة 5: التحقق من معالجة الرسائل والـ Worker
- راقب شاشة الـ Worker في الطرفية (Terminal).
- *التحقق:* يجب أن ترى الـ Worker يلتقط الرسالة من Redis، ويقوم بتصنيف النوايا عبر Gemini، ويدخل البيانات في جدول الـ `Booking` بحالة `PENDING` خلال ثوانٍ.

---

### الخطوة 6: إدارة الحجز من لوحة تحكم الاستقبال
1. افتح صفحة الـ Dashboard المفتوحة مسبقاً.
2. *التحقق:* يجب أن يظهر المريض "عميل جديد" بالرقم `966500000001` فوراً في قائمة المحادثات النشطة.
3. اضغط على المريض للتأكد من ظهور تفاصيل حجزه المستخرجة (الاسم، الخدمة، الطبيب، الفرع، الوقت).
4. اضغط على زر **تأكيد الحجز (Confirm Booking)**.
5. *التحقق:* يجب أن تتغير الحالة في الداتابيز إلى `CONFIRMED` ويظهر البانر الأخضر في الواجهة.

---

### الخطوة 7: اختبار تحمل النظام (Robustness & Recovery)
1. قم بإنهاء عملية الـ Worker (Ctrl+C).
2. أرسل رسالة محاكاة جديدة.
3. *التحقق:* يجب ألا يسقط السيرفر، بل يتم تسجيل الرسالة بنجاح في Redis والانتظار في الطابور.
4. قم بإعادة تشغيل الـ Worker.
5. *التحقق:* يجب أن يقوم الـ Worker بسحب الرسالة فوراً ومعالجتها بدون فقدان أي بيانات.

---

# ✅ Pilot Exit Criteria

لا يعتبر الـ Pilot ناجحًا إلا إذا تحققت جميع البنود التالية:

### Onboarding
- [ ] تم إنشاء عيادة جديدة بالكامل.
- [ ] تم إنشاء الفرع.
- [ ] تم إنشاء الخدمات.
- [ ] تم إنشاء الأطباء (إن كانت جزءًا من الـ Onboarding).
- [ ] لا توجد أخطاء في الـ Logs.

### Authentication
- [ ] تسجيل الدخول نجح.
- [ ] لا يمكن الوصول بدون Login.
- [ ] عزل بيانات كل Tenant يعمل بشكل صحيح.

### WhatsApp
- [ ] الرسالة تصل.
- [ ] الـ Webhook يستقبلها.
- [ ] تدخل الـ Queue.
- [ ] يلتقطها الـ Worker.
- [ ] يرد الـ AI.
- [ ] لا توجد رسائل مفقودة.

### Booking
- [ ] إنشاء حجز.
- [ ] تعديل حجز.
- [ ] إلغاء حجز.
- [ ] تحديث الحالة في Dashboard.
- [ ] تحديث قاعدة البيانات.

### Dashboard
- [ ] جميع الصفحات تفتح.
- [ ] لا توجد Console Errors.
- [ ] لا توجد Buttons معطلة.
- [ ] البيانات محدثة.

### Queue
- [ ] عند إيقاف الـ Worker لا تضيع الرسائل.
- [ ] بعد تشغيله يعالج جميع الرسائل.
- [ ] لا يوجد Duplicate Processing.

### Database
- [ ] لا توجد Duplicate Records.
- [ ] لا توجد Race Conditions.
- [ ] جميع الـ Transactions ناجحة.

### Stability
- [ ] تشغيل وإيقاف الخدمات لا يكسر النظام.
- [ ] إعادة التشغيل لا تسبب فقدان بيانات.

---

### النهاية
- إذا تحققت جميع البنود: **PASS**
- إذا فشل بند واحد: **FAIL**

لا يتم الانتقال لأول عميل تجريبي (Beta Customer) حتى يتم إصلاح سبب الفشل وإعادة الاختبار بنجاح.
